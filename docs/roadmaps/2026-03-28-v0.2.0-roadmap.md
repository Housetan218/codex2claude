# codex2claude v0.2.0 Roadmap

This roadmap captures the highest-value work after the `v0.1.3` release.

## Goals

- make diagnostics more actionable without bloating the bridge
- improve confidence in release quality and edge-case recovery
- keep the Codex skill thin while making daily usage easier

## Planned Work

### 1. Doctor Improvements

- add a dedicated unit test for corrupted thread-state handling in `doctor`
- add human-readable troubleshooting guidance for common `doctor` failure modes
- evaluate a `--plain` mode that summarizes health without requiring JSON reading

### 2. Release Quality

- add a version-consistency check so `pyproject.toml` and `bridge/codex2claude/version.py` cannot drift
- keep release verification reproducible with the isolated packaging environment
- document the exact release checklist used for `v0.1.3`

### 3. Daily Usability

- expand examples for named threads and restart flows
- tighten troubleshooting around skill triggering in fresh Codex sessions
- collect post-release feedback before adding any auto-repair behavior

## Explicit Non-Goals For v0.2.0

- no bidirectional Codex-Claude protocol
- no bridge-managed retry orchestration
- no broad environment doctor that inspects unrelated system state

## Release Gate

`v0.2.0` should only ship after:

- the added diagnostics behavior is covered by tests
- README and CONTRIBUTING stay aligned with actual release steps
- at least one fresh-session natural-language trigger smoke test still passes
